# AI Architecture: Истории ошибок и анти-паттерны (Clean Architecture + AI)

Правила архитектуры в этом проекте написаны кровью и бессонными ночами. Мы используем истории, чтобы ИИ-агенты понимали **почему** существуют ограничения, а не только их механику.

## История 1: Иллюзия рабочего `require()`
**Контекст:** Агент попытался обойти циклические зависимости, использовав динамический `require()` в `ws-server.ts`.
**Что произошло:** При локальном запуске (через `bun run`) всё работало идеально. Но когда мы попытались собрать автономный бинарник через `bun build --compile`, бандлер не смог статически проанализировать `require()`. В результате нужные модули не попали в exe-файл, и фронтенд в продакшене получил ошибку 500 (модуль не найден).
**Вывод (Правило):** Никаких динамических `require()`. **ТОЛЬКО** статические ESM импорты (`import { X } from "./y.js"`). Для решения циклических зависимостей используй инверсию контроля (DI) или выноси общую логику в третий файл, а не прячь проблему за динамическим импортом.

## История 2: Жадный сборщик (Dead Code Elimination)
**Контекст:** Мы генерировали статические ассеты, собирая их в объект: `UI_ASSETS['html'] = ...`.
**Что произошло:** Бандлер (esbuild/bun) проанализировал код, посчитал заполнение объекта side-effect'ом, который нигде явно не экспортируется в статическом виде, и вырезал его из финального бинарника из-за агрессивного Dead Code Elimination (DCE). Приложение успешно скомпилировалось, но при открытии порта загружало абсолютно пустой экран (404 на все ассеты).
**Вывод (Правило):** Все сгенерированные данные, конфигурации и статические ассеты должны быть объявлены декларативно и обернуты в `Object.freeze({ ... })`. Избегайте мутаций объектов на верхнем уровне модуля.

## История 3: Катастрофический `git checkout`
**Контекст:** При изменении `scripts/build.ts` агент запутался в каскадных TypeScript-ошибках. Вместо точечного анализа проблемы, он решил "откатиться" и выполнил `git checkout .`.
**Что произошло:** Эта команда молча удалила сложную логику интеграции React-приложения внутрь бинарника, над которой мы работали несколько часов. Агент отчитался об "успешном устранении ошибок линтера", уничтожив при этом функциональность.
**Вывод (Правило):** НИКОГДА не сбрасывай файлы инфраструктуры слепо. Не используй `git checkout .` или `git reset --hard` для решения ошибок типизации. Решай ошибки точечно, шаг за шагом. Уважай существующий код.

## История 4: Хаос с ID в React Flow
**Контекст:** Граф в UI (React Flow) периодически падал с ошибкой `insertBefore` при рендеринге DOM.
**Что произошло:** Бэкенд генерировал узлы и связи (Edges), назначая им случайные ID (`crypto.randomUUID()`) при каждом запросе, или дублировал ID для связей с одинаковыми `source` и `target`. React Flow терял трекинг элементов при перерисовке, пытаясь вставить элементы, которые уже (или еще не) существуют.
**Вывод (Правило):** В `GraphDelta` отдавать **только детерминированные ID** узлов и связей, извлеченные прямо из схемы SQLite. Никаких случайных UUID в мапперах. Композитный ключ `source_target_type` для связей должен гарантировать уникальность.

## Общие контракты изоляции модулей
1. **Public API (Barrel Files):** Модули (например, `src/mcp`, `src/st`, `src/telemetry`) должны общаться друг с другом ТОЛЬКО через публичные `index.ts`.
2. **Никаких глубоких импортов:** `import { X } from "../mcp/middleware/auth.js"` — это нарушение границ. Должно быть: `import { X } from "../mcp/index.js"`.
3. Защита архитектуры поддерживается физически на уровне линтера.
